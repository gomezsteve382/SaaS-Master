# RFHUB Key-Index Package — 2019 Charger 6.2

This package contains everything extracted from your RFHUB EEPROM dump that is needed to
solve the ONE remaining unknown: the per-key **index byte**. Feed these into your other
software (Autel / VVDI / your own analysis) to reverse the index derivation.

## The problem in one line
To add a brand-new key ID to the RFHUB table offline, each table record needs:
  revUID[4]  index[1]  flag[1=01]  FFFF  (mirrored)
Everything is known EXCEPT the **index** byte. It is not a simple function of the KeyID
or the master secret (brute-forced: xor/add/sub/CRC8 polys 07,1D,31 — zero matches).

## Known-good facts
- SK = MIKRON (4F4E4D494B52) is UNIVERSAL across FCA SRT fobs — confirmed on both the
  working fob and the non-working blank. SK is NOT what pairs a key.
- Empty/unprogrammed slot marker: index=0x95, flag=0x00, revUID=5A5A5A5A.
  Our earlier failed add used index 0x95 (the empty marker) — that's why it didn't take.
- Working keys carry pseudo-random indexes: 0x48,0x0F,0x4C,0x19,0x5B,0xB0.

## What would crack it
1. The same KeyID->index pairs from a SECOND vehicle (different master) — separates the
   key-schedule from the per-vehicle secret.
2. The FCA immobilizer seed-key algorithm (the index is likely a truncated keyed MAC).
3. A bench test of whether ANY non-0x95 index validates (would mean index is just a slot id).
