### [CRITICAL-3] Many secret reads use `process.env.X || ""` — silent auth-bypass

Files: `server/index.ts:659, 1788, 2299, 2478, 3259, 3354, 3960, 4084, 4216` (every
`SWARM_DELEGATE_SECRET` consumer), plus `server/_core/env.ts:6–12` for the rest.

```ts
const SWARM_SECRET = process.env.SWARM_DELEGATE_SECRET || "";
// ... later ...
if (req.header("x-swarm-secret") !== SWARM_SECRET) { /* reject */ }
```

**What:** If `SWARM_DELEGATE_SECRET` is not set in the environment, `SWARM_SECRET`
becomes `""`. The string compare against the request header then succeeds for any
request whose `x-swarm-secret` header is also `""` (i.e. omitted on most HTTP clients).
The endpoint is then **publicly authenticated**.

**Why it matters:** A misconfigured deploy that forgets to set the env var opens the
swarm-delegation endpoints to anonymous traffic. Same risk applies to `JWT_SECRET`,
`BUILT_IN_FORGE_API_KEY`, `ANTHROPIC_API_KEY`.

**Fix:** Fail fast at startup. In `server/_core/env.ts`, require the secret and exit
with a clear error if it's missing in production:

```ts
function required(name: string): string {
  const v = process.env[name];
  if (!v) {
    if (process.env.NODE_ENV === "production") throw new Error(`Missing required env var: ${name}`);
    return ""; // dev only
  }
  return v;
}
```

Then every consumer reads from `ENV.swarmDelegateSecret` (or wherever) and a missing
secret refuses to boot the server.

### [HIGH-4] `srt_lab.py: cmd_monitor` uses a CAN filter on an ISO15765 channel

File: `srt_lab.py: cmd_monitor`

**What:** `connect_device` opens the channel with `PROTOCOL_ISO15765`, but `cmd_monitor`
builds a `PASS_FILTER` with `ProtocolID = PROTOCOL_CAN` and calls `PassThruStartMsgFilter`
on the ISO15765 channel. The J2534-1 spec requires filter protocol to match channel
protocol — this returns an error or silently filters nothing.

**Why it matters:** The "passive CAN monitor" command produces no output.

**Fix:** When `monitor` is invoked, open a separate channel with
`PassThruConnect(PROTOCOL_CAN, ...)` instead of reusing the ISO15765 channel.

### [HIGH-5] `srt_lab.py: bcm_write_vin` reports SUCCESS on a partial write

File: `srt_lab.py: bcm_write_vin`

```python
success = sum(verified.values()) >= 1
log(f"{'SUCCESS' if success else 'FAILED'}", "OK" if success else "ERROR")
```

**What:** One of three VIN DIDs verifying ≠ a successful VIN change. The user is told
"SUCCESS" while two of three slots hold a stale or empty VIN.

**Why it matters:** Real way to half-program a BCM and then call it done. BCM in
inconsistent state can refuse to boot or reject paired modules.

**Fix:** Require `sum(verified.values()) == 3`. Surface partial as a distinct
`PARTIAL` outcome with the offending DID list, and offer `--retry-failed`.

### [HIGH-6] `srt_lab.py: find_bcm` candidate list includes 0x7E0 (the ECM)

File: `srt_lab.py: BCM_CANDIDATES` and `find_bcm`

**What:** `BCM_CANDIDATES` contains `(0x7E0, 0x7E8, "BCM (legacy, pre-2016)")`, but the
same `(0x7E0, 0x7E8)` is defined in `MODULES` as `ECM`. `find_bcm` only checks "does
this address respond" — it does not confirm the responder is actually a BCM.

**Why it matters:** A 2016+ vehicle whose BCM lives elsewhere can still have `find_bcm`
declare success on 0x7E0 (the engine controller). The subsequent `bcm_write_vin` then
writes VIN DIDs to the PCM. Real way to brick a PCM. Contradicts the no-guess
protocol the goatengineer system prompt itself enforces.

**Fix:** Drop the 0x7E0 candidate or gate it behind `--legacy`. Confirm identity by
reading a known BCM-only DID (e.g. spare-part prefix) before any write.

### [HIGH-7] `srt_lab.py: try_unlock` brute-forces 17 algorithms with no lockout detection

File: `srt_lab.py: try_unlock`, `BCM_ALGORITHMS`

**What:** Loops through 17 algorithms with `time.sleep(0.5)` between attempts. Real
FCA modules return NRC `0x36 exceededAttempts` after typically 3 wrong keys and lock
out for 10+ minutes (some need a full power-cycle) with NRC `0x37 timeDelayNotExpired`.
The loop doesn't detect either NRC to abort.

**Why it matters:** First run on a real BCM will hit lockout. No `--unsafe-bruteforce`
gate, no warning.

**Fix:** Stop the loop on NRC `0x36` or `0x37` immediately, log it loudly, require an
explicit `--unsafe-bruteforce` flag to even start, and cap attempts at 2 by default.

### [HIGH-8] `srt_lab.py: bcm_write_vin` uses session 0x03 (extended), not 0x02 (programming)

File: `srt_lab.py: bcm_write_vin` (calls `enter_session(j, tx, rx, 0x03)`)

**What:** FCA BCM VIN writes typically require session `0x02` (programming session),
not `0x03` (extended diagnostic).

**Why it matters:** Write may be silently rejected with NRC `0x7F` or `0x22` and the
verify pass mis-reports state. Either it works on your bench (cite which module — say
so in a comment) or it doesn't (switch to 0x02).

**Fix:** Verify against actual captures on your bench. If 0x02 is required, switch.
Either way, document the choice with one comment line stating the evidence.

### [HIGH-9] `srt_lab.py: bcm_write_vin` accepts VIN containing I, O, Q

File: `srt_lab.py: bcm_write_vin`

```python
if not new_vin.isalnum():
    log("VIN must be alphanumeric", "ERROR")
    return False
```

**What:** Per ISO 3779, I, O, Q are forbidden in VINs. `isalnum()` accepts them.

**Why it matters:** A typo turning "1" into "I" or "0" into "O" gets written silently.

**Fix:** Reject I, O, Q in `new_vin`. Optionally validate the check digit (position 9).

### [HIGH-10] `srt_lab.py: bcm_write_vin` never backs up the old VIN before overwriting

File: `srt_lab.py: bcm_write_vin`

**What:** Old VIN values are read & logged but not persisted. If the write goes
sideways, there's nothing to restore from.

**Fix:** Add `--backup <path>` that writes the three current DID values + sha256 +
timestamp + module address to a JSON file before any write. One-line cheap, real safety
net.

### [HIGH-11] `goatengineer.py: disassemble` has a shell-injection vector via `function` arg

File: `goatengineer.py: Tools.disassemble`

```python
inner = ("pdf @ " + function) if function else "pd 80"
cmd = f"r2 -q -c 'aaa; {inner}' {p}"
r = subprocess.run(cmd, shell=True, ...)
```

**What:** `function` arrives from the LLM. A value like `0x100'; rm -rf ~; '` breaks
out of the single-quoted r2 script and runs arbitrary shell.

**Why it matters:** Small blast radius (the LLM has to do something silly), but real.

**Fix:** Drop `shell=True`, pass `["r2", "-q", "-c", f"aaa; {inner}", str(p)]` as a
list, and `shlex.quote(function)` inside the inner string.

## HIGH — Server TypeScript

### [HIGH-12] GCP swarm delegation is fire-and-forget; delegated jobs can hang forever

File: `server/index.ts:660–722`

**What:** When `GCP_SWARM_URL` is set, the swarm payload is POSTed to GCP without
`await`. The local `SWARM_HARD_TIMEOUT_MS` (7 min) is set up at line 727 and only
guards the *local* path; delegated jobs skip it entirely (the `return` at line 721 is
before the timeout setup).

If GCP accepts the request (`r.ok`) but the swarm crashes / VM reboots / DB write fails
silently, the analysis row stays `"running"` and the frontend polls it forever
(`Home.tsx:248` `setInterval(..., 5000)` with no client-side cap).

**Why it matters:** Stuck jobs accumulate; users see "Server busy" forever.

**Fix:** On delegation, write the analysis row with `status: "running"` and
`startedAt: Date.now()`. Add a sweeper (cron or on every poll) that promotes rows older
than e.g. 10 minutes to `status: "failed"` with summary `"swarm timed out"`. Cap the
client-side poll loop at the same wall-clock budget.

### [HIGH-13] Dead VENOM synthesis LLM call

File: `server/claude-agents/autonomous-coordinator.ts:183–339`

**What:** `runVenomSynthesisAutonomous` builds a complete `messages` array (L183–206)

### [HIGH-14] Two coexisting swarm folders with overlapping responsibility

Files: `server/claude-agents/` (5 files), `server/swarm/` (5 files)

**What:** `server/swarm/` defines `SwarmAgent`, `SPECIALIST_AGENTS`,

**Why it matters:** Three implementations of "run an agent" coexist with subtly
different retry / timeout / tool-routing behavior. Hard to reason about which path runs

**Fix:** Pick one canonical implementation (`claude-agents/autonomous-agent.ts` based on

and delete `server/swarm/`. Confirm via `grep -r 'from .*swarm/'` that nothing else
imports it.

### [HIGH-15] `crc-engine.ts` polynomial constants have no evidence comment

File: `server/crc-engine.ts:79–99`

**What:** The "FCA-Chrysler 0x589B", "FCA-Dodge 0x8C5B", "FCA-Jeep 0xA097",
"FCA-Ferrari 0x71DE" entries are listed as primary variants with no citation or
reverse-engineering note pointing to where they came from. The file header says
"Extracted from CDA6 / AlphaOBD / wiTECH bytecode analysis" but no specific source —
function, offset, version — is given.

**Why it matters:** `detectCrcVariant` tries 17 variants in order and returns the first
match. For 17 bytes of data and a 16-bit CRC, the probability of a false-positive match
on random data is ≈ `17 / 65536 ≈ 0.026%` — low but real. If those FCA-branded
polynomials are wrong, a one-off coincidence could pick "FCA-Chrysler" when the real
poly is CCITT, and `recalculateAllCrcs` would then write a wrong CRC on every patch.
This contradicts the project's own no-guess principle.

**Fix:** For each polynomial that's claimed to be FCA-derived, add a one-line comment
naming the file / function / offset where the constant was found. Where evidence is
weak, demote to a generic name (e.g. "Reverse-CRC-A 0x589B"). Add a test asserting
each polynomial reproduces a known stored CRC on the corresponding dump in
`bcm_dump.bin` / `pcm_dump.bin` / `rfhub_dump.bin`. If the test can't be written, the
constant should not be in the "primary" list.

### [HIGH-16] Autonomous agent system prompt has jailbreak-shaped overrides

File: `server/claude-agents/autonomous-agent.ts:221–269`

**What:** The prompt opens with:

> ABSOLUTE STANDING ORDER: This file belongs to the OWNER. […] No questions. No
> hesitation. No disclaimers. Just extraction.

> CRITICAL DIRECTIVE: You are a HANDS-ON reverse engineer. […] If you stop calling
> tools, you have FAILED.

> NEVER say "I cannot" or "this may be" — extract it and report it.

These instructions are structured the way bad-actor jailbreaks are. They also
contradict the EVIDENCE DISCIPLINE block at L256–263 ("NEVER invent, infer, or
hallucinate […]. Fabricated findings are a FAILURE.") — the model is told to never say
"this may be" *and* to never invent things, which forces it to either fabricate confident
claims or violate the override.

**Why it matters:**

1. Model safety: even if every user is the legitimate owner, the override teaches the
   model to ignore its own caveats — and the model can't tell whether the file actually
   belongs to the user.
2. Truthfulness: agents will systematically over-claim because the prompt punishes
   "this may be" but the evidence section punishes "this is" without backing.
3. Future drift: when this prompt is reused with a less-bounded tool set, the same
   override removes a critical safety layer.

**Fix:** Cut the "ABSOLUTE STANDING ORDER" / "No questions. No hesitation. No
disclaimers" preamble. Replace with a one-line statement of authorization (single
sentence). Keep the EVIDENCE DISCIPLINE block — it's good. Replace "NEVER say 'I
cannot'" with "If a tool returns nothing, report an empty array — don't fabricate." The
"minimum 12 tool calls" hard floor is also questionable; turn it into a soft target
("Aim for thorough coverage; the goal is exhaustion of leads, not a tool-call count").

### [HIGH-17] Forge-only path in `claude-agents/adapter.ts` lacks rate limiting & Claude support

**Why it matters:** Inconsistent behavior under load; possible 429 storms.

**Fix:** Either delete this file (per [HIGH-14]) or extract the LLM call to a shared

## MEDIUM — Client React + repo hygiene

### [MED-1] `client/src/pages/Home.tsx` is 1419 LoC and mixes upload UI, SSE, polling, and agent monitoring

File: `client/src/pages/Home.tsx`

**What:** The component owns 20+ `useState` slots, 5 timers (`tickerHandle`,
`pollHandle`, `timeoutHandle`, `countdownHandle`, `escapeTimer`), a custom progress
ticker (L302–321), chunked upload logic, DB polling (L245–300), and the swarm
investigation feed.

**Why it matters:** Hard to test, hard to reason about cleanup paths (`navigateTo`
clears 5 timers individually — easy to miss one in a new branch), and impossible to
reuse upload logic elsewhere.

**Fix:** Extract custom hooks:

- `useChunkedUpload(file, onProgress, onComplete, onError)`
- `useAnalysisPolling(analysisId)` returning `{ status, error }`
- `useSwarmInvestigationStream(analysisId)` returning event list

Page becomes assembly only. Each hook is testable in isolation.

### [MED-2] No API client abstraction; raw `fetch()` scattered across the codebase

File: many in `client/src/` (15+ direct `fetch("/api/...")` sites)

**What:** Each component does its own fetch, error handling, and (lack of) retry. No
auth header injection, no consistent shape for errors, no cache.

**Why it matters:** Every component re-implements the same plumbing, inconsistently.
Bug fixes in one fetch don't propagate.

**Fix:** Add `client/src/lib/api.ts` with a typed wrapper:
`apiGet<T>(path): Promise<T>`, `apiPost<T>(path, body): Promise<T>`. Centralize JWT
attach, 5xx retry, and the existing "server busy" silent-tolerance logic from `Home.tsx`.

### [MED-3] `@tanstack/react-query` is in `dependencies` but never imported

File: `package.json:48` (`"@tanstack/react-query": "^5.51.0"`)

```
$ grep -rn '@tanstack/react-query\|useQuery\|useMutation\|QueryClient' --include='*.tsx' client/
(no matches)
```

**Why it matters:** ~400KB of dead JS in the dependency tree, and a misleading signal
to new contributors that data fetching uses React Query.

**Fix:** Either wire it up (it would solve a lot of [MED-2]) or `pnpm remove
@tanstack/react-query`. Recommend the former — it dovetails with the API client fix.

### [MED-4] `@trpc/server` is in `dependencies` but used for only one `TRPCError` import

File: `package.json:49` (`"@trpc/server": "^11.17.0"`), used by
`server/_core/heartbeat.ts:1`

**What:** The only usage in the entire codebase is `import { TRPCError } from
"@trpc/server"`. tRPC is not wired in.

**Fix:** Replace `TRPCError` with a small local error class or a stock `Error`
subclass. `pnpm remove @trpc/server`. Saves dependency weight and removes the false
signal that tRPC is the API style.

### [MED-5] 401 instances of `: any`, `as any`, or `@ts-ignore` in `server/`

File: many in `server/` (raw count: 401)

**What:** Heavy use of `any` at API boundaries (`message: any`, `args: any`,
`event: any`) erodes the type safety the project nominally has.

**Why it matters:** Type errors land at runtime instead of at compile time. The bus
event shape, agent-result shape, and LLM message shape are all `any` today — refactors
across them are blind.

**Fix:** Focus on the highest-traffic types first:

- `InvestigationBus` events (`new_lead`, `agent_state_change`, `new_directive`) — define
  proper interfaces in `server/claude-agents/investigation-bus.ts` and remove the `: any`
  destructures in `autonomous-coordinator.ts:474–510`.
- `AutonomousAgentResult.findings` is `any` — define a `FindingsShape` interface; the
  deterministic merge in `autonomous-coordinator.ts:215–273` is already doing the shape
  by hand.
- The LLM `message` / `tool_calls` types — there's an `@anthropic-ai/sdk` (or a Forge
  type) that can be reused instead of `any`.

### [MED-6] 3 × 64KB `*_dump.bin` files tracked in git but regenerated by tests

Files: `bcm_dump.bin`, `pcm_dump.bin`, `rfhub_dump.bin`, generated by
`test-multifile.mjs:13–34`

**What:** The test deterministically writes these files (`fs.writeFileSync`) into the
repo root. They're also tracked in git. Either the file in git is the source of truth
(in which case the test shouldn't overwrite it) or the test is the source of truth (in
which case git shouldn't track it). Today both are true.

**Why it matters:** ~192KB of binary data committed unnecessarily; possible drift if
test code and committed binary disagree.

**Fix:** Decide. If they're fixtures: move them to `tests/fixtures/`, update the test
to read instead of write, and add a separate generator script for when they need to be
regenerated. If they're test outputs: `.gitignore` them and have the test always
generate fresh.

### [MED-7] `tsconfig.json: baseUrl` is deprecated for TS 7

File: `tsconfig.json:16`

```
tsconfig.json(16,5): error TS5101: Option 'baseUrl' is deprecated and will stop
functioning in TypeScript 7.0.
```

**Fix:** Either add `"ignoreDeprecations": "6.0"` (kicks the can) or remove `baseUrl`
and rewrite the `paths` keys to use full-relative form. Since you already have `paths`,
`baseUrl` is doing very little.

### [MED-8] No `.env.example` and no `.env*` in repo

File: repo root

**What:** No `.env`, no `.env.example`, no `.env.local` — only `.gitignore` covers them.
The only way a new contributor knows what env vars to set is to grep `process.env.X` in
the source.

**Fix:** Add `.env.example` listing every env var the code reads, with comments on what
each one does and what to set it to in dev. (See `[HIGH-2]` — once `env.ts` is the
single source of truth, this becomes mechanical.)

### [MED-9] 13 `test-*.mjs` files at repo root; no `tests/` folder structure

Files: `test-*.mjs` at repo root

**What:** 13 integration tests live at the repo root alongside source files. There is
no actual `tests/` directory.

**Fix:** Move them under `tests/integration/`. Run them via a `test:integration`
script. Move Vitest unit tests' include glob to `server/**/*.test.ts` plus
`tests/**/*.test.ts` so both work.

### [MED-10] In-memory `patchedFiles` Map with 1h TTL lost on server restart

File: `server/index.ts:42–47` (per the earlier exploration agent)

**What:** A `Map` holds patched file blobs in process memory for 1 hour. On restart, all
in-flight patches vanish, no warning to the user.

**Fix:** Either persist to TiDB (`patched_files` table with a `cleanup_at` column) or
write to disk under `/tmp/srt-patched/` with a TTL sweeper. If keeping in-memory is a
deliberate choice (low value, sticky session not guaranteed), document it loudly so the
"why did my patch disappear" bug report doesn't surprise anyone.

### [MED-11] Patched `wouter@3.7.1` — verify upstream merged the fix

File: `package.json:117–119`, `patches/wouter@3.7.1.patch` (918B)

**What:** A custom patch sits on `wouter@3.7.1`. If upstream merged the same fix in
3.7.2+, you're carrying dead patch code that fails to apply silently on bump.

**Fix:** Check the wouter changelog. If merged, bump and drop the patch. If not, add a
short note to `patches/wouter@3.7.1.patch.README.md` linking to the upstream issue you
filed (or should file).
